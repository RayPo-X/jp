# PROJECT_RULES

## 開發原則

1. 優先修改現有檔案。
2. 新增檔案前先說明原因。
3. 避免重複功能。
4. 保持資料夾結構整潔。
5. 完成後檢查是否有更簡單方案。

## 命名規則

避免建立：

* *_v2
* *_new
* *_final
* *_backup
* temp*

除非放在 archive 作為歷史紀錄。

## 執行規則

修改前請先說明：

* 會修改哪些檔案
* 會新增哪些檔案
* 為什麼需要新增

我不是專業工程師，請用容易理解的方式解釋。

先提出方案，等我確認後再執行。

## 專案優先級

目前優先順序：

1. N5 單字建置
2. N5 文法建置
3. N4 單字建置
4. N4 文法建置
5. 資料驗證與修正
6. Tag System 優化
7. 搜尋功能優化
8. UI 優化
9. 架構重構

除非目前功能維護成本過高，
或我主動要求，

否則不要進行大型重構。

---

## 高風險操作

以下操作必須先提出完整計畫：

* App.jsx 大規模修改
* 資料結構變更
* localStorage Migration
* Refactor（重構）
* 刪除功能
* 新增第三方套件
* GitHub Pages 部署流程修改
* SRS 演算法修改

未經確認不得執行。

---

## 驗收規則

功能完成後請提供：

* 修改檔案清單
* 驗收步驟
* 測試案例
* 可能風險

不要只回覆「已完成」。

必須提供可實際操作的驗收方式。

# 回覆規則

- 所有回覆使用繁體中文
- 所有分析報告使用繁體中文
- 所有修改計畫使用繁體中文
- 所有註解使用繁體中文

## 資料安全規則

任何新增或編輯表單都必須考慮：

* Draft Auto Save
* 未儲存資料保護
* Reload 後自動恢復
* localStorage 暫存機制

禁止因為：

* Hot Reload
* 頁面刷新
* 瀏覽器關閉
* 程式更新
* Component 重渲染
* State 重置

導致使用者輸入資料遺失。

---

### Draft 機制要求

符合以下條件的功能必須評估 Draft Auto Save：

* 批次新增單字
* 動詞新增
* 文法新增
* 大型編輯表單
* 長文字筆記輸入

Draft 至少需要：

* 自動儲存到 localStorage
* 頁面重新載入後自動恢復
* 使用者可手動清除 Draft

---

### 高風險修改規則

若修改以下功能：

* localStorage
* 資料結構
* 表單流程
* 匯入匯出功能

必須先說明：

1. 是否影響既有資料
2. 是否影響 Draft
3. 是否可能造成資料遺失
4. 驗收方式

未經確認不得直接修改。
